## RAG-агент поддержки TrainBot

Модуль реализует RAG-агента поддержки TrainBot на базе LlamaIndex. Агент предназначен для автоматического ответа на пользовательские вопросы о работе TrainBot: подписках, бронировании, уведомлениях, возвратах, аккаунте РЖД, пассажирах, тарифах и типовых ошибках.

Система построена не как простой поиск по базе знаний, а как проверяемый RAG-пайплайн. Каждый ответ формируется на основе найденных источников, после чего дополнительно проверяется на релевантность вопросу и подтверждённость найденными контекстами.

### Источники знаний

В системе используется несколько источников данных:

1. **Q&A-база поддержки TrainBot**

   Краткая база типовых пользовательских обращений и канонических ответов службы поддержки. Она лучше подходит для практических вопросов: возврат средств, ID пользователя, проблемы с подписками, ошибки бота, отсутствие брони, уведомления, пассажиры и аккаунт РЖД.

2. **PDF-документация TrainBot**

   Документация с подробными инструкциями по разделам сервиса. Она используется для вопросов, где нужен пошаговый сценарий: создание подписки, настройка параметров, работа с разделами интерфейса и другие инструкции.

3. **VerifiedAnswerMemory**

   Отдельное Chroma-хранилище проверенных ответов. В него сохраняются только те ответы, которые успешно прошли проверки `RelevancyEvaluator` и `FaithfulnessEvaluator`. При следующих похожих вопросах система сначала проверяет это хранилище и может вернуть уже проверенный ответ без повторного запуска полного RAG-пайплайна.

Q&A-база и PDF-документация подключаются как Chroma-векторные хранилища. Для всех источников используется общая модель эмбеддингов HuggingFace, чтобы поиск выполнялся в едином векторном пространстве.

### Общий алгоритм работы

Работа агента организована через `FaithfulRAGWorkflow`.

На вход workflow получает исходный пользовательский вопрос. Перед поиском вопрос нормализуется: из него удаляются лишние приветствия, эмоции, повторы и несущественные подробности, но сохраняются важные данные — сумма, дата, ID, тип подписки, ошибка, списание денег, отсутствие брони и другие значимые детали. Нормализованный вопрос используется как поисковый запрос.

После нормализации система сначала обращается к `VerifiedAnswerMemory`. Если там найден похожий ранее проверенный ответ с достаточно малой дистанцией, workflow сразу возвращает этот ответ со статусом `cached`. Полный запуск агента, поиск по источникам и проверки в этом случае не выполняются.

Если подходящего ответа в памяти нет, запускается основной `ReActAgent`. Агент получает исходный вопрос пользователя и нормализованный поисковый запрос. По системному промпту он обязан использовать инструмент `main_engine_tool` и отвечать только на основе найденных фрагментов.

### Маршрутизация между источниками

Инструмент `main_engine_tool` является главным поисковым инструментом TrainBot. Он реализован как `FunctionTool`, потому что ему нужен доступ к `workflow Context`.

Внутри `main_engine_tool` вызывается `RouterQueryEngine`. Для выбора источника используется `LLMSingleSelector`, который определяет, какой из двух инструментов лучше подходит для конкретного вопроса:

* `rag_tool` — поиск по краткой Q&A-базе поддержки TrainBot;
* `pdf_document_tool` — поиск по PDF-документации TrainBot.

Например, если пользователь спрашивает о возврате денег или проблеме с бронированием, чаще выбирается Q&A-база. Если вопрос требует инструкции по оформлению подписки, настройки параметров или работы с разделом интерфейса, выбирается PDF-документация.

После выбора источника `RouterQueryEngine` выполняет поиск, получает ответ и возвращает `source_nodes`. Context-aware обёртка `main_engine_tool` сохраняет в `Context`:

* выбранный источник;
* найденные контексты `source_nodes`;
* сырой ответ RAG-инструмента;
* количество найденных контекстов.

Эти данные затем используются workflow для проверки ответа.

### Проверка ответа

После ответа агента workflow выполняет две проверки.

Сначала запускается `RelevancyEvaluator`. Он проверяет, соответствует ли ответ исходному вопросу пользователя и релевантны ли найденные контексты.

Если проверка релевантности пройдена, запускается `FaithfulnessEvaluator`. Он проверяет, подтверждается ли ответ найденными источниками. Это нужно, чтобы агент не добавлял неподтверждённые причины, правила, тарифы, сроки возврата или технические детали.

Если обе проверки пройдены, workflow возвращает ответ пользователю со статусом `passed`. Дополнительно успешный ответ сохраняется в `VerifiedAnswerMemory`, чтобы его можно было повторно использовать при похожих вопросах.

### Повторная попытка поиска

Если ответ не прошёл проверку, workflow может выполнить повторную попытку.

При retry система не вызывает конкретный источник напрямую. Вместо этого она меняет формулировку поискового запроса так, чтобы `RouterQueryEngine` попробовал другой поисковый ракурс.

Например:

* если сначала был выбран `rag_tool`, следующая попытка смещается в сторону документации TrainBot;
* если сначала был выбран `pdf_document_tool`, следующая попытка смещается в сторону похожих пользовательских обращений.

Список уже использованных источников хранится в `tried_sources`. Это позволяет остановить повторные попытки, если оба источника уже были проверены и релевантного ответа всё равно не найдено.

### Возможные статусы результата

Workflow может вернуть несколько вариантов результата:

* `cached` — найден готовый проверенный ответ в `VerifiedAnswerMemory`;
* `passed` — агент сформировал ответ, и он прошёл проверки релевантности и подтверждённости;
* `partially_supported` — ответ релевантен вопросу, но не все формулировки строго подтверждены источниками;
* `needs_clarification` — в базе знаний TrainBot не найдено точной информации, пользователю нужно уточнить вопрос.

Если точной информации нет, агент не должен придумывать ответ. В этом случае возвращается безопасная формулировка: в базе знаний TrainBot нет точной информации по вопросу, и пользователю предлагается уточнить детали.

### Итоговая схема

Упрощённо алгоритм выглядит так:

```text
Пользовательский вопрос
        ↓
Нормализация вопроса
        ↓
Поиск в VerifiedAnswerMemory
        ↓
Если найден проверенный ответ → вернуть cached-ответ
        ↓
Если не найден → запуск ReActAgent
        ↓
main_engine_tool
        ↓
RouterQueryEngine
        ↓
Выбор источника: Q&A-база или PDF-документация
        ↓
Получение ответа и source_nodes
        ↓
Сохранение контекстов в workflow Context
        ↓
RelevancyEvaluator
        ↓
FaithfulnessEvaluator
        ↓
Если проверки пройдены → сохранить ответ в VerifiedAnswerMemory и вернуть пользователю
        ↓
Если проверки не пройдены → retry с другим поисковым ракурсом
        ↓
Если точного ответа нет → безопасный fallback
```

Таким образом, модуль реализует полноценный проверяемый RAG-пайплайн для поддержки TrainBot. Он не только ищет информацию в базе знаний, но и нормализует пользовательский вопрос, использует память проверенных ответов, автоматически выбирает подходящий источник, сохраняет найденные контексты, проверяет качество ответа и выполняет повторную попытку поиска при неудачной проверке.
